                    ISSUE 4: Duplicate Subject Prevention Timing
================================================================================
SEVERITY: MEDIUM (Potential race condition)
FILE: src/services/hybrid/subject.service.ts

PROBLEM:
  The duplicate check via SOAP and subsequent creation has a race condition:
  
  1. Check if subject exists (SOAP call)
  2. Wait for response
  3. Create subject
  
  Between steps 1 and 3, another request could create a subject with the same label.
  
CURRENT CODE (lines 94-114):
  try {
    const subjectExists = await subjectSoap.isSubjectExists(...);
    if (subjectExists) {
      return { success: false, message: 'Subject already exists' };
    }
  } catch { ... }
  
  // GAP HERE - Subject could be created by another request
  
  const result = await subjectSoap.createSubject(...);

FIX REQUIRED:
  Use database unique constraint to catch duplicates instead of relying on
  pre-check. The code already handles this via error code '23505' but should
  be the primary mechanism, not fallback.

================================================================================
                    ISSUE 5: Event Date Calculation May Be Off
================================================================================
SEVERITY: LOW (Cosmetic)
FILE: src/services/hybrid/subject.service.ts

PROBLEM:
  Event start dates are calculated as enrollment date + (ordinal - 1) * 7 days.
  This is a simple offset, but:
  
  1. Doesn't account for weekends/holidays
  2. Hardcoded 7-day interval may not match study protocol
  3. No configuration for interval per event type

CURRENT CODE (lines 322-324):
  const daysOffset = (eventDef.ordinal - 1) * 7;
  const eventStartDate = new Date(enrollmentDate);
  eventStartDate.setDate(eventStartDate.getDate() + daysOffset);

RECOMMENDATION:
  Make interval configurable per study or event definition.
  Consider adding study_event_definition.days_from_enrollment column.

                    ISSUE 6: Frontend Gender Value Inconsistency
================================================================================
SEVERITY: MEDIUM (Data mapping issue)
FILE: ElectronicDataCaptureReal/src/app/components/patient-enrollment-modal/patient-enrollment-modal.component.ts

PROBLEM:
  Frontend sends 'male'/'female' but backend expects 'm'/'f'.
  
CURRENT CODE (lines 590-598):
  let normalizedGender: string | undefined;
  if (formValue.gender === 'male' || formValue.gender === 'm') {
    normalizedGender = 'm';
  } else if (formValue.gender === 'female' || formValue.gender === 'f') {
    normalizedGender = 'f';
  } else if (formValue.gender) {
    normalizedGender = '';
  }

ISSUE:
  This is correct, BUT the form uses radio buttons with values 'male' and 'female'
  (lines 242-245 in HTML), not the backend format.
  
  If the backend is updated to accept 'male'/'female', these conversions would
  double-convert and break.

RECOMMENDATION:
  Standardize on backend format ('m'/'f') in frontend form values to eliminate
  conversion complexity.

                    ISSUE 7: Audit Log Event Type Lookup May Fail
================================================================================
SEVERITY: MEDIUM (Audit logging might not work)
FILE: src/services/hybrid/subject.service.ts

PROBLEM:
  Audit log event type is looked up by name pattern matching:
  
CURRENT CODE (lines 269-274):
  await client.query(`
    INSERT INTO audit_log_event (...)
    VALUES (...,
      (SELECT audit_log_event_type_id FROM audit_log_event_type 
       WHERE name LIKE '%Create%' OR name LIKE '%Insert%' LIMIT 1))
  `);

ISSUE:
  - LIKE '%Create%' may return wrong event type
  - If no match found, inserts NULL for audit_log_event_type_id
  - Should use specific known values from audit_log_event_type table

FIX REQUIRED:
  Use specific event type IDs:
  - 1 = 'Entity Created'
  - 2 = 'Entity Updated'
  - 3 = 'Entity Deleted'
  
  Or better, define constants and use them:
  const AUDIT_EVENT_TYPES = {
    ENTITY_CREATED: 1,
    ENTITY_UPDATED: 2,
    ENTITY_DELETED: 3,
    ...
  };
